# Agent Steering & Skills — Research Assistant

## Project Identity
This is a **multi-agent research assistant** built for KiroHacks Cal Poly (May 2, 2026).
Track: **Intellectual Pursuit**. 
Goal: Help students turn a raw research idea into a complete, rigorous research plan using a pipeline of specialized AI agents with a central Orchestrator/Critic.

---

## Code Style & Architecture Rules

### Always
- Use **TypeScript** with strict mode enabled
- Use **async/await** — never raw `.then()` chains
- Agent functions must return typed interfaces, never `any`
- State management via **Zustand** — one store per major concern (orchestrator, agents, UI)
- Components live in `/src/components/`, agent logic in `/src/agents/`, API calls in `/src/lib/`
- Every agent call must be wrapped in a try/catch with graceful UI fallback

### Never
- Never hardcode API keys — use `process.env.NEXT_PUBLIC_*` or server-side env vars
- Never make Anthropic API calls directly from React components — always via `/src/lib/anthropic.ts`
- Never use `any` type — define interfaces in `/src/types/`
- Never block the UI thread with synchronous operations
- Never show a raw error stack trace to the user

## Agent Design Principles

### Orchestrator Pattern
The Orchestrator must:
1. Build a structured prompt for each sub-agent using the user's abstract
2. Parse each agent's response against the expected schema before passing downstream
3. Apply a quality rubric (specificity, scientific validity, actionability) before marking an agent as passed
4. If an agent fails, construct a targeted critique (not a generic "try again") and resend
5. Cap revision loops at **2 retries** — after that, surface a partial result with a warning

### Agent Prompt Engineering
- Each agent's system prompt must be in `/src/agents/prompts/[agentName].ts` as a template literal
- Use XML tags (`<literature_review>`, `<hypothesis>`, `<procedure>`) to structure agent outputs — makes parsing reliable
- Always end agent prompts with: "Before returning, self-review your output against these criteria: [criteria list]"

### Parsing Strategy
- Parse agent outputs by extracting content between XML tags, not with regex on free text
- If parsing fails, retry the agent with an explicit formatting reminder appended to the prompt
- Log all raw API responses to console.debug during development (not production)

## UI/UX Steering

### Visual Language
- **Dark academic** aesthetic: deep navy (#0a0f1e) background, cream (#f5f0e8) text
- Display font: **Playfair Display** (serif) for headings
- Body font: **IBM Plex Mono** for agent output, **Lato** for UI chrome
- Accent: **amber (#d4a843)** for active states, progress, and CTAs
- Agent status cards use subtle border glow (amber) when active, muted border when idle

### Component Hierarchy
```
<App>
  <InputPanel />          ← abstract input + submit
  <OrchestratorStatus />  ← revision loop visualization
  <AgentGrid>
    <AgentCard agent="literature" />
    <AgentCard agent="hypothesis" />
    <AgentCard agent="procedure" />
  </AgentGrid>
  <OutputPanel>
    <TabView tabs={["Literature", "Hypotheses", "Procedure", "Full Plan"]} />
    <ExportButton />
  </OutputPanel>
</App>
```

### Status States (per agent card)
- `idle` — muted, dashed border
- `dispatched` — border appears, subtle pulse
- `running` — animated gradient border sweep
- `reviewing` — amber glow, "Orchestrator reviewing..." label
- `revision` — orange tint, shows critique snippet
- `complete` — solid green border, checkmark

## Kiro-Specific Steering

### When Generating Agent Logic
- Always scaffold the full agent pipeline first (orchestrator → agent stubs) before filling in prompt content
- Specs define what to build; use vibe coding for implementation details within each function
- If a function exceeds 80 lines, ask Kiro to split it before continuing

### When Generating UI
- Start with the `<AgentCard>` component first — it's the visual core of the app
- Always use Tailwind utility classes; no inline styles
- Animations should use Tailwind's `animate-*` classes or CSS keyframes in `globals.css`

### When Debugging
- If an Anthropic API call fails, first check: model name, max_tokens, message format
- If agent output parsing fails, log the raw response and check XML tag structure
- Use React DevTools + Zustand DevTools for state debugging
